# Development Log — April 15, 2026

## Summary

This session focused entirely on building and refining the **Data Agent** — a natural language interface embedded in the dashboard that lets users describe customer interactions, deals, and notes in free-form text (English or Turkish), which the system parses, clarifies, and executes as structured database operations.

---

## What Was Built

### 1. Data Agent Infrastructure

**New file: `data_agent_orchestrator.py`**

A standalone orchestration module implementing a 5-phase pipeline:

1. **Parse** — Sends the user's raw message to Ollama (`qwen2.5:14b`) with a structured prompt, producing a JSON list of intended actions.
2. **Resolve Entities** — Fuzzy-matches customer names against the DB. Progressive matching strategy: exact → LIKE → strip possessive ('s) → drop last word.
3. **Check Required Fields** — Validates NOT NULL constraints, fetches the matching deal row for `update_deal` actions, and builds a clarification queue for anything missing.
4. **Build Action Cards** — Creates human-readable "display cards" for each action (title, details list, icon, confirm/skip flag).
5. **Execute** — Writes to `CustomerDeals` or `Comment` tables; for `update_deal`, only overwrites fields that are actually changing (overlay strategy).

**Key constants:**
- `CURRENCY_MAP` / `CURRENCY_LABELS`: `{try:0, usd:1, eur:19}`
- `STATUS_CODES` / `STATUS_LABELS`: `{Lead:1, Proposal:2, Due Diligence:3, Closed Won:4, Closed Lost:5}`
- `DEALTYPE_LABELS`: `{1:Syndication, 2:Bahrain, 3:Sukuk, 4:KT AG}` — stored as NVARCHAR in DB, cast to `int()` for lookup
- `_cache_current(action, row)` — stores current DB values onto the action dict so the confirmation card can show accurate "from → to" change summaries

**Supported action types:**

| Type | Trigger | Required Fields |
|------|---------|----------------|
| `insert_deal` | New deal mentioned | `customer_name` |
| `insert_comment` | Meeting, call, note, follow-up | `customer_name`, `comment_text` |
| `update_deal` | Change to existing deal, or implied status change | `customer_name` |
| `query_deals` | List/check existing deals | `customer_name` |

`update_deal` has two field categories:
- **Filter fields** (`filter_dealtype`, `filter_status`, `filter_contact`, `filter_size_approx`): identify *which* deal to target
- **Update fields** (`dealtype_hint`, `status_hint`, `deal_size`, `currency`, `contact_name`, `notes`): what to change

`query_deals` auto-executes without user confirmation (`requires_confirmation: False`).

### 2. Flask Endpoints

**New routes in `app.py`:**

- `POST /api/agent/chat` — Main state machine endpoint. Manages `session['agent_state']` across turns.
  - Phases: `idle → clarifying → confirming → done`
  - Handles typed overrides and chip-pick answers during clarification
  - Auto-executes `requires_confirmation=False` actions
  - Returns `all_cards` list so the frontend can show all actions at once

- `POST /api/agent/reset` — Clears agent session state.

**`_ask_confirmation()` response shape:**
```json
{
  "phase": "confirming",
  "current_action": { ... },
  "action_index": 0,
  "total_actions": 2,
  "all_cards": [ {...}, {...} ]
}
```

### 3. Intent Parser Prompt

**New file: `knowledge_base/intent_parser_logic.txt`**

The LLM prompt governing all intent extraction. Key rules:

- **RULE 5 — Status Inference (semantic, not keyword):** The LLM reasons about what a described action implies about deal stage, rather than matching against a fixed phrase list. E.g., "completed the offer and sent it" → Proposal. Handles arbitrary natural language and Turkish inputs.

- **RULE 8 — Dual-action pattern:** When a communication event also implies a status change (e.g., "we sent them the offer"), the LLM must produce TWO actions — an `insert_comment` AND an `update_deal` with the inferred `status_hint`. This prevents the agent from only logging the comment and missing the status update.

- **RULE 10 — Filter vs Update:** A descriptor *before* the word "deal" is always a filter (identifies which deal), not an update. E.g., "sukuk deal completed" → `filter_dealtype: Sukuk`, `status_hint: Closed Won`.

### 4. Dashboard UI — Data Agent Panel

**Modified: `templates/dashboard.html`**

The chatbox was replaced with a full multi-turn agent UI:

**Message log:** Shows user messages (right-aligned, brand colour) and agent responses (left-aligned, styled by type — amber for questions, blue for confirmations, green for done).

**Clarification phase:** Displays a question with clickable chip options. Typing while in clarifying phase sends the text as an answer (not a new message) — controlled by `agentPhase` JS variable.

**Review panel (confirming phase):** All pending actions are shown together in a single card, so the user always sees the full picture. Layout:
- Already handled actions: greyed out with ✓ Confirmed or → Skipped label
- Current active action: highlighted in blue, full details, Confirm + Skip buttons inline
- Upcoming actions: dimmed with numbered indicators

This replaced the previous one-at-a-time confirm bar where "Skip" felt like abandoning the entire flow.

**Done phase:** Renders a summary of all results (success/skip/fail per action). `query_deals` results render as an inline table.

## Bugs Fixed During Session

| Bug | Root Cause | Fix |
|-----|-----------|-----|
| `conn.close()` before `check_required_fields` | DB connection was being closed before it was passed to the required-fields checker | Moved `conn.close()` to after Phase 4 |
| `KeyError: 'notes'` / HTTP 500 | `SELECT` in `check_required_fields` was missing the `notes` column | Added `notes` to the SELECT |
| `dealtype` label showed wrong value | `dealtype` is stored as NVARCHAR (`"1"`, `"2"`…), but `DEALTYPE_LABELS` keys are ints | Applied `int()` cast before dict lookup |
| `check_required_fields` TypeError | Missing `conn=None` default parameter | Added `conn=None` to signature |
| Form submit during clarifying started a new turn | JS always sent `{message: ...}` regardless of phase | JS now sends `{action:'answer', answer:...}` when `agentPhase === 'clarifying'` |
| "ABCs" not resolved (possessives) | Fuzzy matcher had no possessive-strip step | Added progressive fuzzy: exact → LIKE → strip `'s` → drop last word |
| "sukuk deal completed" not applying filter | `dealtype_hint` was extracted as update field, not filter | Added RULE 10 + LLM fallback: use `dealtype_hint` as filter when other update fields exist |
| from→to display showed filter values, not real DB values | `_cache_current` wasn't being called | Added `_cache_current(action, row)` call in `check_required_fields` |
| Agent only logged comment, missed status update | LLM not instructed to split communication events into dual actions | Added RULE 8 dual-action pattern + expanded TYPE 3 trigger description |
| "offer sent" not recognised as Proposal | RULE 5 was a rigid phrase list | Rewrote RULE 5 as a semantic reasoning instruction |

## Architecture Notes

### Session State (`session['agent_state']`)
```python
{
  "phase":         "idle|clarifying|confirming|done",
  "actions":       [...],        # resolved action dicts
  "results":       [...],        # accumulated execution results
  "confirm_index": 0,            # which action is next to confirm
  "clarif_queue":  [...],        # pending clarification questions
  "clarif_index":  0,
}
```

### DB Schema (relevant tables)
```sql
CustomerDeals (id, customerid, contact_name NOT NULL, deal_size NOT NULL,
               expected_pricing_pa, currency INT NOT NULL, status INT NOT NULL,
               dealtype NVARCHAR, notes, created_at NOT NULL)

Comment (customer_id, author, content)
```

### Ollama Config
- Model: `qwen2.5:14b`
- URL: `http://localhost:11434` (env: `OLLAMA_URL`)
- Timeout: 120s (env: `DATA_AGENT_TIMEOUT`)
